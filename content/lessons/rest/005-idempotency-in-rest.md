# Idempotency in REST

We touched on Idempotency in the Distributed Systems and Kafka modules, but it is a first-class requirement of the REST specification.

## 1. The HTTP Specification

The HTTP/1.1 specification strictly mandates the idempotency of specific methods:

* **`GET`, `PUT`, `DELETE`**: MUST be Idempotent. (Calling them 1 time or 100 times must leave the server in the exact same state).
* **`POST`**: Is NOT Idempotent. (Calling it 100 times creates 100 resources).

## 2. The Network Problem

Imagine an iOS app calls `POST /payments` to charge a user $100.
The Go server receives the request, talks to Stripe, successfully charges the credit card, and sends a `201 Created` response back.

**The Crash:** A router drops the packet. The iOS app never receives the `201 Created`. It times out. 

The user clicks "Pay" again. The iOS app sends a second `POST /payments`. The Go server charges the credit card a second time. The user is furious.

Because `POST` is not idempotent, standard network retries are catastrophically dangerous!

## 3. The Idempotency-Key Header

Enterprise companies (like Stripe) solve this by forcing clients to provide a unique **Idempotency-Key** HTTP Header on every `POST` request.

`Idempotency-Key: 8a9b2c-4d5e-6f7a` (Usually a UUID generated by the iOS app).

### The Go Implementation

1. The Go server receives the `POST` request and reads the Header.
2. The Go server checks a fast datastore (like Redis or PostgreSQL).
3. `SELECT response_body FROM idempotency_keys WHERE key = '8a9b2c...'`
4. **First Request (Cache Miss)**: 
   * The Go server processes the payment.
   * The Go server saves the resulting JSON response to Redis, linked to that Idempotency Key.
   * Returns the `201 Created`.
5. **The Retry Request (Cache Hit)**:
   * The iOS app retries using the exact same Idempotency Key.
   * The Go server checks Redis. It finds the key!
   * The Go server **aborts the business logic**. It simply fetches the old JSON response from Redis and returns it to the client!

The iOS app thinks the second request succeeded natively, and the user's credit card is only charged exactly once!

## 4. PUT vs POST (When to use which?)

Many developers use `POST` for creating resources, and `PUT` for updating them. 
This is generally correct, but technically inaccurate.

You can use `PUT` to **Create** a resource, IF the client dictates the ID!

* `POST /users`: The server generates the ID (e.g., `42`). This is NOT idempotent.
* `PUT /users/42`: The client tells the server, "Create a user here, and make their ID 42". 

If the network drops, and the client retries `PUT /users/42`, the Go server just overwrites User 42 with the exact same data. It is perfectly idempotent, requiring no `Idempotency-Key` headers or Redis caching!
