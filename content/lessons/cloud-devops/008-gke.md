# Google Kubernetes Engine (GKE)

## 1. Learning Objectives
* **What you'll learn**: How to deploy and manage enterprise-grade Go applications on Google Kubernetes Engine (GKE), leveraging Autopilot and Workload Identity.
* **Why it matters**: Managing a raw Kubernetes control plane (API server, etcd) is a DevOps nightmare. GKE abstracts the Master Nodes away completely, letting you focus purely on deploying your Go code while Google handles the cluster infrastructure.
* **Where it's used**: Massive microservice deployments, stateful databases, and complex ML workloads at companies like Spotify and Twitter.

---

## 2. Real-world Story
Imagine owning a factory. 
If you build raw Kubernetes on your own servers (Bare Metal), you have to build the walls, wire the electricity, and fix the plumbing every time a pipe bursts.
If you use GKE, Google builds the factory, wires the electricity, and maintains the plumbing 24/7. You simply walk in, place your Go application machines on the factory floor, and press "Start". You get all the power of Kubernetes with zero of the administrative headaches.

## 4. Internal Working (Under the Hood)
GKE offers two distinct modes:
1. **Standard Mode**: You manage the underlying physical VMs (Node Pools). If you want 10 servers, you rent 10 servers. If your Go Pods only use 10% of the RAM, you are wasting money on the other 90%.
2. **Autopilot Mode**: The modern standard. You do not manage VMs. You just submit a Deployment YAML for your Go app requesting `512Mi` of RAM. Google automatically provisions the exact microscopic sliver of infrastructure required, and you only pay for that `512Mi`. It is "Serverless Kubernetes".

---

## 5. Compiler Behavior
* **Container Scanning**: Because GKE is deeply integrated into Google Cloud, when you push your compiled Go Docker image to Google Artifact Registry, Google natively scans the binary for known CVEs. If your Go binary contains a vulnerable package, GKE can optionally block the deployment at the API level!

---

## 6. Memory Management
* **Right-Sizing Pods**: In GKE Autopilot, setting correct CPU/Memory requests is critical because you are billed exactly on what you request. A standard Go microservice often needs only `cpu: 250m` (1/4th of a core) and `memory: 128Mi`. Over-provisioning (like requesting 2GB of RAM for a Go app) is a massive waste of cloud budget.

## 7. Code Examples

### 🔹 Example 1: The GKE Ingress
```yaml
# To expose your Go service to the internet in GKE, you use an Ingress.
# GKE magically watches this YAML and automatically provisions a 
# physical Google Cloud HTTP(S) Load Balancer in your GCP console!
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: goverse-ingress
spec:
  rules:
  - host: api.goverse.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: goverse-service
            port:
              number: 80
```

### 🔹 Example 2: Workload Identity (Security)
```yaml
# NEVER hardcode AWS/GCP Service Account keys in your Go code or K8s Secrets!
# GKE Workload Identity allows your K8s ServiceAccount to seamlessly act as a GCP IAM Role.
apiVersion: v1
kind: ServiceAccount
metadata:
  name: goverse-sa
  annotations:
    iam.gke.io/gcp-service-account: my-gcp-role@my-project.iam.gserviceaccount.com
```

### 🔹 Example 3: Connecting to Cloud SQL from Go (Securely)
```yaml
# To connect your Go Pod to a private Google Cloud SQL database,
# you run the Cloud SQL Auth Proxy as a "Sidecar" container in the exact same Pod!
      containers:
      - name: go-backend
        image: myrepo/goverse-api
        env:
        # The Go app connects to localhost! The proxy forwards it securely to GCP.
        - name: DB_HOST
          value: "127.0.0.1" 
      
      - name: cloud-sql-proxy
        image: gcr.io/cloud-sql-connectors/cloud-sql-proxy:2.1.0
        args:
          - "--private-ip"
          - "my-project:us-central1:my-instance"
```

### 🔹 Example 4: Production (Google Managed SSL)
```yaml
# GKE can automatically generate and renew Let's Encrypt SSL Certificates for you!
apiVersion: networking.gke.io/v1
kind: ManagedCertificate
metadata:
  name: goverse-cert
spec:
  domains:
    - api.goverse.com
```

### 🔹 Example 5: Interview
```go
// Q: Why would you choose GKE over Cloud Run (Serverless) for a Go application?
// A: Cloud Run is purely for stateless HTTP traffic. If you need a persistent gRPC stream, 
// a stateful Redis cluster, background workers that run indefinitely, or complex internal 
// networking (Service Mesh), you MUST use GKE.
```

## 8. Production Examples
1. **Multi-Zone Clusters**: GKE can distribute your Go Pods across three different physical data centers (e.g., `us-central1-a`, `b`, `c`). If a meteor hits Data Center A, the K8s Master instantly routes all traffic to B and C. The user never notices.
2. **Prometheus Integration**: GKE has "Managed Prometheus" built-in. You simply add an annotation to your Go Pod, and Google automatically scrapes your `/metrics` endpoint without you needing to install the heavy Prometheus server yourself.

---

## 9. Performance & Benchmarking
* **VPC Native Networking**: GKE uses Alias IPs. Your Go Pod gets a real IP address from the Google Cloud VPC network natively. This allows your Go Pod to talk directly to a Google Compute Engine VM or an AWS server via VPN with zero NAT (Network Address Translation) latency overhead.

---

## 10. Best Practices
* ✅ **Do**: Use Workload Identity. It is the only secure way to grant your Go application permission to read from Google Cloud Storage or Cloud SQL without managing raw JSON keys.
* ❌ **Don't**: Expose internal Go microservices via `LoadBalancer` services. This creates a public IP and charges you $15/month per service. Use `ClusterIP` for internal communication.
* 🏢 **Google / Uber / Netflix Style**: Use **Anthos / GKE Enterprise**. It allows you to manage GKE clusters, AWS EKS clusters, and on-premise servers from a single pane of glass using a unified Service Mesh (Istio).

---

## 11. Common Mistakes
1. **The Default Node Pool**: Using the default node pool for production. The default pool cannot be easily upgraded without downtime. Always create a custom node pool! (Or just use Autopilot).
2. **Ignoring Preemptible VMs**: If your Go background workers process non-critical batch jobs, you can use Preemptible/Spot VMs in GKE Standard. Google will randomly kill your VMs every 24 hours, but they cost 80% less money!

---

## 12. Debugging
How to troubleshoot GKE in production:
* **Cloud Trace**: If you use OpenTelemetry in your Go code, GKE natively exports those traces to Google Cloud Trace. You can view a waterfall diagram of an HTTP request entering the GKE Ingress, hitting the Go Pod, and executing a Postgres query.

## 14. Quiz
1. **MCQ**: What is the primary difference between GKE Standard and GKE Autopilot?
   * (A) Standard uses Docker, Autopilot uses VMs. (B) Autopilot manages the Worker Nodes and billing at the Pod level, eliminating VM management. (C) Standard cannot run Go code. *(Answer: B)*
2. **System Design Follow-up**: How does the GKE Ingress controller know which Go Pods to route traffic to if the Pod IPs change constantly? *(The Ingress Controller talks directly to the Kubernetes API. It maintains a dynamic map of Endpoint IPs. When a pod dies and a new one is created, the API instantly updates the Ingress Controller's routing table).*

## 17. Enterprise Features & Observability
* **ConfigMap Sync**: Instead of rebuilding your Go Docker image just to change an environment variable, store variables in a K8s `ConfigMap`. GKE mounts the ConfigMap into your Pod as a file or env var. If you update the ConfigMap, the Go app can hot-reload the changes!

---

## 18. Source Code Reading
Walkthrough of `google.golang.org/api`.
* **The Google Go SDK**: Study how the official Google SDK natively understands GCP metadata servers. If the Go app is running on GKE, the SDK detects it via HTTP calls to `metadata.google.internal` and automatically fetches short-lived Oauth2 tokens!

---

## 19. Architecture
* **Hub and Spoke**: For massive enterprises, you don't run 1 giant GKE cluster. You run 50 small GKE clusters (one per team). You connect them via a Hub-and-Spoke VPN architecture. This limits the "Blast Radius" if a cluster's control plane completely fails.

---

## 20. Summary & Cheat Sheet
* **Master Node**: Hidden and managed by Google.
* **Autopilot**: Pay per Pod, no Node management.
* **Workload Identity**: Securely access GCP APIs.
* **Ingress**: Provisions a physical Google Load Balancer.
