# Google Kubernetes Engine (GKE)

While Cloud Run is amazing for simple web APIs, enterprise architectures often require background workers, gRPC internal meshes, daemonsets, and complex persistent storage. For this, you need a full Kubernetes cluster.

**GKE (Google Kubernetes Engine)** is arguably the best managed Kubernetes service in the world (Google invented Kubernetes).

## 1. GKE Autopilot vs Standard

GKE offers two flavors:
1. **Standard**: You provision the underlying virtual machines (Nodes). You decide if you want 5 huge VMs or 10 small VMs. You pay for the VMs whether they are full or empty.
2. **Autopilot**: You don't manage Nodes at all. You simply submit your Go app's `deployment.yaml` to the cluster, and GKE provisions the exact CPU and RAM required to run your Pods. You pay per-Pod. This is essentially "Serverless Kubernetes".

## 2. Workload Identity (The Security Revolution)

Historically, if your Go application running inside Kubernetes needed to read a file from a Google Cloud Storage bucket, you had to download a highly sensitive JSON Service Account Key, save it as a Kubernetes Secret, and mount it into your Pod.

If a hacker accessed that JSON file, they had keys to your entire cloud.

**Workload Identity** fixes this entirely. It binds a Kubernetes Service Account directly to a Google Cloud IAM Service Account using OIDC (OpenID Connect) trust.

1. You create a Google IAM Account: `storage-reader@my-project.iam.gserviceaccount.com`.
2. You grant it read access to the S3 Bucket.
3. You bind it to a Kubernetes Service Account.
4. You assign the Kubernetes Service Account to your Go Pod in the `deployment.yaml`.

```yaml
apiVersion: apps/v1
kind: Deployment
spec:
  template:
    spec:
      serviceAccountName: my-k8s-service-account # Binds to IAM!
      containers:
      - name: go-api
        image: my-go-api:v1
```

Now, the Go SDK inside your application uses "Application Default Credentials". It automatically asks the GKE metadata server for a short-lived OAuth token. **No JSON keys are ever generated, stored, or mounted.** 

## 3. Horizontal Pod Autoscaling (HPA)

Because Go binaries are so efficient, they boot instantly. This makes GKE autoscaling incredibly responsive.

You can configure an HPA to watch the CPU usage of your Go app. If CPU goes above 70%, it will automatically clone your Pods.

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: go-api-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: go-api-deployment
  minReplicas: 2
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
```
When traffic spikes, GKE will rapidly scale your Go application from 2 pods up to 10 pods to handle the load, and then scale back down to 2 when traffic subsides to save money!
