# Introduction

In a monolithic application, if a user's request fails, you simply look at the stack trace in your terminal. It tells you exactly which function on which line of code crashed.

In a microservices architecture, a single user request might travel through an API Gateway, an Auth Service, a Billing Service, and a Database across 4 different physical servers. If the request takes 5 seconds, how do you know which of the 4 services is the slow one? 

**Distributed Tracing** solves this. It attaches a unique ID to a request the moment it enters your system, and passes that ID along to every downstream service. This allows you to stitch the entire journey together into a single, visual timeline.

# Why This Topic Exists

Imagine a user submits a "Checkout" request. It takes 10 seconds and returns a 500 Error.
You look at the logs for the `CheckoutService`. It says: `Waiting for PaymentService... Error: Timeout`.
You look at the logs for the `PaymentService`. There are 5,000 logs from that exact second. Which log belongs to the user who failed? You have no idea. You are flying blind.

With Distributed Tracing, the API Gateway generates a `Trace ID: abc-123`. It passes `abc-123` to the CheckoutService, which passes it to the PaymentService. You search `abc-123` in your tracing UI, and it instantly shows you a visual waterfall chart of exactly how many milliseconds the request spent in each service, and exactly where it crashed.

# Core Concepts: Traces and Spans

### 1. Trace ID
A globally unique identifier (e.g., a UUID) representing a single user's journey through the entire system. Every log and metric generated by that user's request across all microservices will share this identical Trace ID.

### 2. Span ID
A Span represents a single, logical unit of work within a Trace (e.g., "A Database Query", or "An HTTP request to the Auth Service"). A single Trace contains multiple Spans. Spans have a Start Time, an End Time, and a Parent Span ID. 

This Parent-Child relationship is what allows tracing UIs to build nested Waterfall charts.

---

# Context Propagation

How does the PaymentService know what the Trace ID is? 
**Propagation**.
1. **In-Process**: Within your Go application, you store the Trace ID inside the `context.Context` and pass the context to every function.
2. **Cross-Process**: When making an HTTP request to another microservice, you extract the Trace ID from the context and inject it into the HTTP Headers (usually using the W3C standard header `traceparent`). The receiving Go server parses the header and puts it back into its own `context.Context`.

# Step-by-Step Implementation (OpenTelemetry in Go)

In the past, developers used libraries specifically for Jaeger or Datadog. Today, the industry has standardized on **OpenTelemetry (OTel)**, a vendor-neutral API.

1. Import `go.opentelemetry.io/otel`.
2. Configure a Global Tracer Provider (tells the app where to send the span data, e.g., to Jaeger).
3. At the entry point of your HTTP handler, start a span: `ctx, span := tracer.Start(req.Context(), "HandlerName")`.
4. Defer ending the span: `defer span.End()`.
5. Pass the `ctx` to all downstream functions.
6. When calling external APIs, use the `otelhttp` middleware, which automatically injects the Trace ID into the outbound HTTP Headers.

# Beginner Example

Instrumenting a simple Go function. (Requires `go get go.opentelemetry.io/otel`).

```go
package main

import (
	"context"
	"fmt"
	"time"

	"go.opentelemetry.io/otel"
)

// Initialize a global tracer named "my-app"
var tracer = otel.Tracer("my-app")

func ProcessOrder(ctx context.Context) {
	// 1. Start a new Span. It automatically links to any parent span in the ctx!
	ctx, span := tracer.Start(ctx, "ProcessOrder")
	
	// 2. GUARANTEE the span ends when the function returns
	defer span.End()

	fmt.Println("Processing order...")
	time.Sleep(100 * time.Millisecond)

	// Pass the context down to the next function
	ChargeCard(ctx)
}

func ChargeCard(ctx context.Context) {
	// This span is automatically a CHILD of "ProcessOrder"
	_, span := tracer.Start(ctx, "ChargeCard")
	defer span.End()

	fmt.Println("Charging card...")
	time.Sleep(50 * time.Millisecond)
	
	// You can add metadata to spans!
	span.AddEvent("Card Charged Successfully")
}

func main() {
	// In a real app, this context comes from the HTTP Request
	ctx := context.Background()

	// Start the Root Span
	ctx, rootSpan := tracer.Start(ctx, "CheckoutTransaction")
	defer rootSpan.End()

	ProcessOrder(ctx)
	
	fmt.Println("Trace Complete. (In production, this data is sent to Jaeger).")
}
```

# Production Use Cases

### 1. Root Cause Analysis (Jaeger / Datadog)
When an alert fires saying "Latency on API Gateway is 5 seconds", an engineer opens Jaeger, filters for traces exceeding 4 seconds, and clicks one. The UI instantly displays a waterfall chart showing that the API Gateway took 5 seconds *because* the Database Span took 4.9 seconds. The engineer instantly knows to check the Database, bypassing hours of reading logs.

### 2. Identifying N+1 Database Bugs
When you view a trace in a UI, you might see 1 HTTP Span, and then 50 tiny, sequential SQL Spans inside it. This visually exposes an "N+1 Query" bug (doing a separate database query in a `for` loop instead of a single batch query). Tracing is the fastest way to visualize poorly optimized code.

# Best Practices

* **Always pass `ctx`**: Distributed tracing is the absolute biggest reason why idiomatic Go code *requires* passing `context.Context` as the first argument to every single function. If you break the context chain, the Trace ID is lost, and the trace is broken.
* **Tag Spans with useful data**: Use `span.SetAttributes()` to add metadata (e.g., `user_id`, `http.status_code`). This allows you to search your tracing UI for "All traces where user_id = 123".
* **Don't trace everything**: In massive systems, exporting a span for 100,000 requests per second will DDOS your tracing backend (and cost millions of dollars in Datadog bills). Use **Sampling**. Configure OpenTelemetry to only export 1% of successful traces, but 100% of error traces.

---

# Common Mistakes

### Forgetting to inject headers
If Service A calls Service B using a standard `http.Client{}`, it doesn't matter if you pass the context to the `http.NewRequestWithContext`. The Trace ID remains trapped in Go memory. You *must* inject the Trace ID into the HTTP headers so it travels across the network. (Use `otelhttp.Transport` to do this automatically).

# Quiz

## Multiple Choice Questions
**1. How does Service B know the Trace ID generated by Service A?**
A) Service B queries a central database for the Trace ID.
B) Service A serializes the Trace ID from its local `context.Context` and injects it into the HTTP headers (e.g., `traceparent`) of the outgoing network request. Service B parses the header.
C) Service A uses RPC to send it.
*Answer*: B

## True or False
**A single Trace contains exactly one Span.**
*Answer*: False. A Trace represents the entire journey. A Span represents a specific operation within that journey. A single Trace typically contains dozens or hundreds of nested Spans.

# Interview Questions

## Beginner
**Q**: What is the difference between Distributed Tracing and Logging?
*Answer*: Logging records individual, isolated events that happened on a specific server (e.g., "User logged in"). Distributed Tracing stitches together the entire lifecycle of a request across *multiple* servers, showing exactly how long each step took and how they relate to each other temporally.

## Intermediate
**Q**: Why is `context.Context` so critical to implementing Distributed Tracing in Go?
*Answer*: `context.Context` acts as the invisible backpack for the request. When the OpenTelemetry middleware creates a Span, it stores the Span ID inside the context. By passing the context to every function call, inner functions can retrieve the active Span ID to link themselves as "children", constructing the tree-like trace structure without using global variables.

## Advanced
**Q**: Explain Head-Based vs Tail-Based Sampling in Distributed Tracing.
*Answer*: When dealing with massive scale, exporting 100% of traces is too expensive. 
* **Head-Based Sampling**: The API Gateway decides right when the request begins whether to sample it (e.g., roll a 100-sided die; if it's 1, trace it). It's very fast, but you might accidentally drop traces that later crash and become interesting.
* **Tail-Based Sampling**: Every service generates spans and sends them to a collector buffer. The collector waits until the entire trace is finished. If the trace had an error or took longer than 5 seconds, it saves it permanently. Otherwise, it throws it away. It guarantees you capture 100% of errors, but requires massive memory buffers on the collector.

# Summary

Distributed Tracing shines a flashlight into the dark, complex web of microservices. By rigorously passing `context.Context` and adopting OpenTelemetry, you transform chaotic network calls into beautiful, actionable waterfall charts, drastically reducing the Mean Time To Resolution (MTTR) during a production outage.

# Key Takeaways

* ✔ Trace ID = The entire request journey.
* ✔ Span ID = A single unit of work (e.g., a DB query).
* ✔ Trace contexts must be injected into HTTP Headers to cross networks.
* ✔ OpenTelemetry is the vendor-neutral industry standard.
* ✔ Never break the `context` chain in your Go code.
