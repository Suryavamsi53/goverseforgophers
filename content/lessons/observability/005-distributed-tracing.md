# Distributed Tracing

In a monolithic application, if a function is slow, you run a CPU profiler (`pprof`) to find the bottleneck. 

But in a modern architecture, a single user request might travel through an API Gateway, an Auth Service, an Order Service, and a Postgres Database. If the request takes 5 seconds, **which service is to blame?** 

You cannot use a CPU profiler across 4 different servers. You must use **Distributed Tracing**.

## 1. TraceIDs and Spans

A Trace represents the entire journey of a request. It is composed of **Spans**.
* **TraceID**: A globally unique identifier (e.g., a UUID) assigned to the request the moment it enters the API Gateway.
* **SpanID**: A unique identifier for a specific operation (e.g., "Validate User" or "Query Database").
* **ParentSpanID**: The ID of the span that called this span, forming a parent-child relationship.

## 2. Context Propagation (W3C Trace Context)

How does the Order Service know what the `TraceID` is when it is called by the Auth Service?

The TraceID must be propagated over the network. The industry standard is the W3C Trace Context HTTP header: `traceparent`.

```http
GET /orders HTTP/1.1
Host: order-service.internal
traceparent: 00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01
```
When your Go web server receives this request, the OpenTelemetry middleware automatically intercepts the header, extracts the `TraceID`, and injects it into the `r.Context()`. 

Whenever you make a downstream HTTP request to the Database, the OTel HTTP Client automatically extracts the TraceID from the Context and injects it into the outbound headers!

## 3. Visualizing with Jaeger

The Spans generated by your Go applications are exported to a tracing backend like **Jaeger** or **Datadog APM**.

These tools stitch the Spans together using their Parent/Child relationships and generate a massive waterfall visualization.

```mermaid
gantt
    title Jaeger UI Visualization (TraceID: 4bf92f...)
    dateFormat  s
    axisFormat %S
    
    section API Gateway
    HTTP GET /checkout : 0, 5s
    
    section Auth Service
    Validate Token : 0s, 1s
    
    section Order Service
    Calculate Tax : 1s, 1s
    Save Order : 2s, 3s
    
    section Postgres
    INSERT INTO orders : 2s, 3s
```
Looking at this graph, the engineer can instantly see that the API Gateway is not the problem. The bottleneck is the `INSERT INTO orders` span inside the Postgres database taking 3 seconds! 

Without Distributed Tracing, this debugging process could take weeks of analyzing disjointed text logs across multiple teams.
